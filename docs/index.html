Heterogeneous Treatment Effects

Causal machine learning for CATE and CRR in one R package

hte3 helps practitioners estimate how treatment effects vary across people. It offers a production-style workflow for conditional average treatment effects (CATE) and conditional risk ratios (CRR), while still exposing the lower-level sl3-based machinery when you want tighter control.

In practice, that means you can start with a simple three-step flow and only drop into advanced learners, stacks, and tuning when your analysis needs it.

What hte3 is for

  • Estimating effect heterogeneity instead of only one average effect.
  • Working with effect modifiers and confounders in a clean task object.
  • Choosing between simple baselines and more flexible meta-learners.
  • Keeping an approachable user path while staying in the sl3 ecosystem.

Quick Start

library(hte3)

task <- hte_task(
  data = hte3_example_data(),
  modifiers = c("W1", "W2", "W3"),
  confounders = c("W1", "W2", "W3"),
  treatment = "A",
  outcome = "Y",
  cross_fit = FALSE
)

fit <- fit_cate(task, method = "dr")
pred <- predict(fit)

What does the package actually estimate?

Think of hte3 as a package for learning effect surfaces, not just one-number summaries.

CATE

Conditional average treatment effects

Use CATE when your question is "for which types of units is treatment more or less helpful?" The output is a conditional mean difference, and the main entry point is fit_cate(). 

fit_cate(task, method = "dr")
CRR

Conditional risk ratios

Use CRR when a relative effect is more natural than a difference and the outcome is nonnegative. The output is a conditional risk-ratio surface, and the main entry point is fit_crr(). 

fit_crr(task, method = "ep")
Practitioner takeaway: start by choosing the estimand that matches your reporting goal, then choose the method inside that estimand.

Why hte3

User-friendly on the surface, flexible underneath

The package is designed so you can stay in a high-level workflow for common analyses, while still having access to the lower-level learners and tuning controls that advanced users expect.

Production-style workflow

Build an hte3_Task once with hte_task(), then fit with fit_cate() or fit_crr().

Method choice stays visible

You do not have to guess which meta-learner is active. The site and the API keep DR, R, EP, T, and IPW workflows explicit.

sl3-native customization

If you want custom nuisance learners, stacks, selectors, or cross-fitting behavior, the package stays close to the sl3 idioms you already know.

      Workflow

      The day-to-day workflow is intentionally short

      Most analyses can start with three simple decisions: what data object to build, which estimand to target, and which learner to plug in.

      Step 1

      Build a task

      Use hte_task() to define modifiers, confounders, treatment, and outcome. The task object keeps the analysis inputs explicit.

      Step 2

      Fit the estimand

      Call fit_cate() or fit_crr(), then choose the method that matches your analysis goal and comfort level.

      Step 3

      Predict and compare

      Use predict() and summary() to inspect the fitted surface, then compare simpler and richer methods when needed.

      Tip: if you are not sure what learner library to use, keep the default get_autoML() stack at first and tune later.
